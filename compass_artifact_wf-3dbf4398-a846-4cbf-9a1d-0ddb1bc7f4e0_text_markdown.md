# Mercado Livre (MLB) API — Complete Endpoint & Field Reference for an Enterprise Orders + Shipping Cockpit

## TL;DR
- The full cockpit is buildable today against `api.mercadolibre.com` using six resource families — Orders, Packs, Shipments (with `x-format-new: true`), Post-Purchase claims/returns, Billing, and Notifications — with per-logistic-type branching driven by `shipment.logistic.type` (fulfillment / cross_docking / self_service / xd_drop_off / drop_off / custom) and `shipment.logistic.mode` (me1/me2/custom/not_specified).
- Real-time state comes from webhooks (`orders_v2`, `shipments`, `messages`, `claims`, `fbm_stock`/`fbm_stock_operations`, `invoices`, `flex_handshakes`); every notification only carries a `resource` URL (confirmed payload: `{"resource":"/orders/2195160686","user_id":...,"topic":"orders_v2","application_id":...,"attempts":1,"sent":...,"received":...}`), so each event requires a follow-up GET. `/orders/search` is capped at 50 records/call and does NOT support `search_type=scan`.
- Brazil-specific: fiscal (NF-e) readiness gates shipping — Full auto-invoices via ML's biller, while cross_docking/xd_drop_off/fulfillment forbid seller invoice upload; the "Adicione os dados fiscais" state maps to shipment substatus `invoice_pending`/`awaiting_tax_documentation` and the `/can_invoice/items/{id}` + `/orders/{id}/billing_info` resources.

## Key Findings

### Authentication & scopes (all areas)
- OAuth2 Authorization Code (server-side), optional PKCE. Token endpoint `POST https://api.mercadolibre.com/oauth/token`. Scopes: `read`, `write`, `offline_access`. `offline_access` is required to obtain a `refresh_token`.
- `access_token` valid exactly 6 hours (`expires_in: 21600`; 200 body: `{"access_token","token_type":"Bearer","expires_in":21600,"scope":"offline_access read write","user_id","refresh_token":"TG-..."}`); `refresh_token` single-use, valid up to 6 months; only the last refresh token is valid.
- All reads (orders, shipments, billing, reputation) need `read`; state changes (label print flows, messages, claim actions, tracking updates, fiscal writes) need `write`. Token becomes invalid if the app is idle 4 months, user changes password, or app secret refreshes.
- App-level rate cap `max_requests_per_hour` (18000 by default on the sample app). Flex resources: 1000 rpm hard limit. `/marketplace/orders/search` (CBT): 100 req/min.

### 1. ORDERS API
- **Search**: `GET /orders/search` (standard) — must include a role filter (`seller=$SELLER_ID` or `buyer=$BUYER_ID`). Verbatim filter keys: `seller`, `buyer`, `order.status`, `order.date_created.from`, `order.date_created.to`, `order.date_closed.from`, `order.date_closed.to`, `order.date_last_updated.from`, `order.date_last_updated.to`, `tags`, `tags.not`, `q` (order id / item id / item title / counterpart nickname — NOT first_name/last_name/email), `item`, `mediations.stage`, `mediations.status`, `feedback.status`, `feedback.sale.rating`, `feedback.sale.fulfilled`, `feedback.purchase.rating`, `feedback.purchase.fulfilled`. Sort param `sort` = `date_asc` (default) | `date_desc` (sellers sort by date_closed, buyers by date_created). Pagination: `offset`, `limit` (default/observed max 50). Date filters use hour granularity only.
  - `manually_cancelled` status is visible only when searching as buyer; seller searches filter cancelled orders. Orders retained ~12 months.
  - **Scan/scroll is NOT supported on `/orders/search`** — only `/users/{id}/items/search` and `/questions/search` support `search_type=scan` + `scroll_id` (expires 5 min): *"To search more than 1000 records of Items, Questions and Answers... Send the parameter search_type=scan to the query and release the offset."* For orders beyond 50, page via `offset` + date windows (note: `offset` has a hard maximum of 10,000 records; for very large listings use date-window/from_id partitioning).
  - CBT variant `GET /marketplace/orders/search` returns a wrapper with `results[].orders[].id` (use the inner `orders` node id, not the top id) and supports sorts `date_asc/date_desc/updated_asc/updated_desc/closed_asc/closed_desc`.
- **Get one**: `GET /orders/{order_id}`. Top-level fields: `id`, `status`, `status_detail`, `date_created`, `date_closed`, `last_updated`, `manufacturing_ending_date`, `pack_id`, `pickup_id`, `fulfilled`, `buying_mode`, `shipping_cost`, `total_amount`, `paid_amount`, `currency_id`, `coupon{id,amount}`, `expiration_date`, `mediations[]`, `comments`, `cancel_detail`, `order_request{return,change}`, `context{channel,site,flows}`, `tags[]`.
  - `order_items[]`: `item{id,title,category_id,variation_id,seller_custom_field,variation_attributes[],warranty,condition,seller_sku,global_price,net_weight,user_product_id,release_date,parent_item_id}`, `quantity`, `requested_quantity{measure,value}`, `picked_quantity`, `unit_price`, `full_unit_price`, `gross_price`, `currency_id`, `sale_fee`, `listing_type_id`, `manufacturing_days`, `base_exchange_rate`. (`gross_price = (unit_price + discounts.full) × quantity`.)
  - `payments[]`: `id`, `order_id`, `payer_id`/`collector{id}`, `status`, `status_detail`, `status_code`, `transaction_amount`, `total_paid_amount`, `taxes_amount`, `shipping_cost`, `coupon_amount`, `overpaid_amount`, `installments`, `installment_amount`, `date_approved`, `date_created`, `date_last_modified`, `payment_method_id`, `payment_type`, `operation_type`, `authorization_code`, `card_id`, `issuer_id`, `available_actions[]`, `site_id`, `reason`.
  - `buyer{id,nickname,first_name,last_name,billing_info{doc_type,doc_number}}` (contact fields obfuscated per privacy), `seller{id,nickname}`, `feedback{sale,purchase}`, `shipping{id}`.
  - Partial responses return HTTP 206 with header `X-Content-Missing` listing missing nodes (buyer/feedback/mediations/seller/shipping).
  - Order sub-resources: `GET /orders/{id}/product` (IMEI/entry_date attributes), `GET /orders/{id}/discounts`, `GET /orders/{id}/billing_info` (x-version:2), `GET /orders/{id}/shipments`, `GET /orders/{id}/feedback`, `POST /orders/{id}/feedback`.
- **Order tags** (seller/system): `paid`, `not_paid`, `delivered`, `not_delivered`, `pack_order` (auto for cart orders, cannot be removed), `catalog`, `fraud_risk_detected` (fraud alert → cancel), `test_order`. Note: `delivered`/`not_delivered` are NO LONGER added automatically — the integrator must PUT them.
- **Cancel groups** in `cancel_detail`: `mediations`, `fiscal`, `buyer`, `fraud`, `item`, `shipment`, `delivery`, `seller`, `internal`.

### 2. PACKS / carts
- `GET /packs/{pack_id}` → `{id, status, status_detail, buyer{id}, date_created, orders[]{id}, shipment{id}}`. 1 pack → N orders (1:N); pack → shipping 0:1; order → payments 1:N. Query `/packs` first for overview, then `/orders/{id}` per order.
- Pack status includes `released`; when a cart is split, pack `status=cancelled`, `status_detail=splitted`.
- `GET /packs/{pack_id}/fiscal_documents` → `{pack_id, fiscal_documents[]{id,date,file_type,filename}}`.

### 3. SHIPMENTS API (Mercado Envios) — send `x-format-new: true` on all GETs
- **Get shipment**: `GET /shipments/{shipment_id}` → `id`, `external_reference` (= pack number), `status`, `substatus`, `date_created`, `last_updated`, `declared_value`, `currency_id`, `dimensions{height,width,length,weight}`, `logistic{direction(forward/return),mode(me2/me1/custom/not_specified),type(drop_off/cross_docking/self_service/xd_drop_off/fulfillment/turbo/custom)}`, `source{site_id,market_place,application_id}`, `tracking_number`, `tracking_method`, `origin{type,sender_id/node,shipping_address{...}}`, `destination{type,receiver_id,receiver_name,receiver_phone(obfuscated),comments,shipping_address{...}}`, `lead_time{...}`, `tags[]`, `snapshot_packing{snapshot_id,pack_hash}`, `items_types[]`, `buffering{date}`.
  - `shipping_address` fields: `id`, `address_id`, `address_line`, `street_name`, `street_number`, `comment`, `zip_code`, `city{id,name}`, `state{id,name}`, `country{id,name}`, `neighborhood{id,name}`, `municipality{id,name}`, `types[]` (billing/default_return_address/default_selling_address/flex_pickup/shipping), `agency{carrier_id,agency_id,description,phone,type,open_hours}`, `latitude`, `longitude`, `geolocation_type`, `geolocation_source`, `is_valid_for_carrier`. `destination.shipping_address` is the new equivalent of the old `receiver_address`; obfuscated until payment confirmed.
- **Sub-resources**:
  - `GET /shipments/{id}/items` → `[{item_id,description,quantity,variation_id,dimensions{...},order_id,sender_id}]`.
  - `GET /shipments/{id}/costs` → `gross_amount`, `receiver{user_id,cost,compensation,save,discounts[]{rate,type,promoted_amount}}`, `senders[]{user_id,cost,compensation,save,discounts[]}`. (`gross_amount` = total cost with no discount; `senders[].cost` = final cost charged to the seller.)
  - `GET /shipments/{id}/payments` → `[{payment_id,user_id,amount,status}]`.
  - `GET /shipments/{id}/lead_time` → `option_id`, `shipping_method{id,type,name,deliver_to}`, `currency_id`, `cost`, `list_cost`, `cost_type`(free/charged/partially_free), `service_id`, `delivery_type`, `estimated_schedule_limit{date}`, `buffering{date}`, `estimated_delivery_time{type,date,shipping,handling,unit,offset{date,shipping},time_frame{from,to},pay_before,schedule}`, `estimated_handling_limit{date}`, `estimated_delivery_extended{date}`, `estimated_delivery_limit{date}`, `estimated_delivery_final{date}`, `delay[]`.
    - `estimated_handling_limit` = seller dispatch deadline (date only); `estimated_delivery_extended` = 2nd delivery promise; `estimated_delivery_limit` = buyer refund-cancel deadline; `estimated_delivery_final` = final date to determine delivered/not_delivered.
  - `GET /shipments/{id}/history` → `[{status,substatus,date}]`.
  - `GET /shipments/{id}/carrier` → `{url,name}` (e.g. `{"url":"http://tracking.totalexpress.com.br/...","name":"Total Express"}`).
  - `GET /shipments/{id}/sla` → `{status(on_time/delayed/early), service(e.g. xd_same_day), expected_date, last_updated}`. Do not call for cancelled or Fulfillment shipments.
  - `GET /shipments/{id}/delays` → `{shipment_id, delays[]{type(handling_delayed/sla_delayed),date,source}}`; 404 "Delays Not Found" when on time.
  - `GET /shipment_statuses` → full status/substatus catalog.
  - `POST /shipments/{id}/split` (x-format-new) — not for Flex/Full; `reason` ∈ FRAGILE/ANOTHER_WAREHOUSE/IRREGULAR_SHAPE/OTHER_MOTIVE/DIMENSIONS_EXCEEDED; body `packs[]{package_id?,orders[]{id,quantity}}`.
- **Full status → substatus list** (from `/shipment_statuses`):
  - `pending`: cost_exceeded, under_review, reviewed, fraudulent, waiting_for_payment, shipment_paid, creating_route, manufacturing, buffered, creating_shipping_order.
  - `handling`: regenerating, waiting_for_label_generation, invoice_pending, waiting_for_return_confirmation, return_confirmed, manufacturing, agency_unavailable.
  - `ready_to_ship`: ready_to_print, invoice_pending, printed, in_pickup_list, ready_for_pkl_creation, ready_for_pickup, ready_for_dropoff, picked_up, stale, dropped_off, delayed, claimed_me, waiting_for_last_mile_authorization, rejected_in_hub, in_transit, in_warehouse, ready_to_pack, in_hub, measures_ready, waiting_for_carrier_authorization, authorized_by_carrier, in_packing_list, in_plp, on_hold, packed, on_route_to_pickup, picking_up, shipping_order_initialized, looking_for_driver.
  - `shipped`: delayed, waiting_for_withdrawal, contact_with_carrier_required, receiver_absent, reclaimed, not_localized, forwarded_to_third, soon_deliver, refused_delivery, bad_address, changed_address, negative_feedback, need_review, stale, operator_intervention, claimed_me, retained, out_for_delivery, delivery_failed, waiting_for_confirmation, at_the_door, buyer_edt_limit_stale, delivery_blocked, awaiting_tax_documentation, dangerous_area, buyer_rescheduled, failover, picked_up, dropped_off, at_customs, delayed_at_customs, left_customs, missing_sender_payment, missing_sender_documentation, missing_recipient_documentation, missing_recipient_payment, import_taxes_paid.
  - `delivered`: damaged, fulfilled_feedback, no_action_taken, double_refund, inferred.
  - `not_delivered`: returning_to_sender, receiver_absent, to_review, destroyed, waiting_for_withdrawal, negative_feedback, not_localized, double_refund, cancelled_measurement_exceeded, returned_to_hub, returned_to_agency, picked_up_for_return, claimed_me, returning_to_warehouse, returning_to_hub, soon_to_be_returned, return_failed, in_storage, pending_recovery, agency_unavailable, rejected_damaged, refused_delivery, refunded_by_delay, delayed, delayed_to_hub, shipment_stopped, awaiting_tax_documentation, retained, stolen, returned, confiscated, damaged, lost, recovered, returned_to_warehouse, not_recovered, detained_at_customs, detained_at_origin, unclaimed, import_tax_rejected, import_tax_expired, rider_not_found.
  - `cancelled`: recovered, label_expired, cancelled_manually, fraudulent, return_expired, return_session_expired, unfulfillable, closed_by_user, pack_splitted, shipped_outside_me, shipped_outside_me_trusted, inferred_shipped, service_unavailable, dismissed, time_expired, pack_partially_cancelled, rejected_manually, closed_store, out_of_range.
  - Also `to_be_agreed`, `not_verified`, `closed`, `error`, `active`, `not_specified`, `stale_ready_to_ship`, `stale_shipped` (no substatuses).
- **Front↔API mapping** (cross_docking): "Em preparação"=handling; "A caminho"=ready_to_ship (substatus picked_up/authorized_by_carrier); "Recebido no CD"=ready_to_ship+in_hub; "Entregue"=delivered. Fulfillment: "Em preparação"=handling; "A caminho"=shipped; "Entregue"=delivered.
- **ME1 / custom tracking updates**: `POST /shipments/{id}/seller_notifications` with `{payload{service_id,comment,date},tracking_number,tracking_url,status,substatus}`. `service_id` per site — MLB: 11, MLA: 154, MLM: 231876, MLC: 282578, MCO: 282579, MLU: 282604, MPE: 361180. Statuses shipped/not_delivered(returning_to_sender)/delivered(null) are finalizing. Custom shipping: `PUT /shipments/{id}` with `{speed,status,tracking_number,receiver_id}`.

### 4. SHIPPING LABELS
- `GET /shipment_labels?shipment_ids=$ID1,$ID2&response_type=pdf` (PDF) or `&response_type=zpl2` (ZPL/Zebra → returns a ZIP with a PDF PLP + Zebra TXT). Max 50 shipment_ids per call (400 error if exceeded). Available only when shipment `status=ready_to_ship`. Standard label size 15×10 cm (México 20×10 cm); PDFs reducible to 9×15 to fit three per A4.
- CBT: `GET /marketplace/shipments/{id}/labels` (PDF).
- **Full**: sellers CANNOT print sale labels — ML does it (stock is in ML warehouses). Drop-off/cross_docking/xd_drop_off: seller prints when `ready_to_ship`; if substatus `buffered`, check the `buffering.date` node for when the label becomes printable. Brazil: NF-e prints at same dimensions and only after generation, independently or with the label.

### 5. FLEX (self_service)
- Identify: category `shipping_preferences.logistics[].types[]` contains `self_service`; item shipping `logistic_type=self_service`, tags include `self_service_in`; check item Flex offering `GET /flex/sites/{site_id}/items/{item_id}/v2`.
- Driver/assignment: `GET /flex/sites/{site_id}/shipments/{shipment_id}/assignment/v1` — identifies the driver assigned to a shipment (tracks driver changes). AR: off-site collections require the seller's alphanumeric authorization code (Settings → Selling preferences → Authorization codes).
- Delivery windows/cutoffs: `PUT /shipping/flex/sites/{site_id}/users/{user_id}/services/{service_id}/configuration/delivery/custom/v3` with `{delivery_window(same_day),delivery_ranges{week[]{from,to,capacity,cutoff},saturday[],sunday[]}}`. `from`/`to` = shipping window hours, `capacity` = delivery limit, `cutoff` = order cutoff time.
- Subscriptions: endpoint to check user's Flex/Turbo subscriptions (Turbo requires Flex active first). Deactivate item: `DELETE /sites/{site}/shipping/selfservice/items/{item_id}`.
- Notifications: subscribe to `flex_handshakes` for package handoffs between drivers and first scan. The Flex driver app is required to scan/route and is NOT available to integrations. 1000 rpm limit.

### 6. FULFILLMENT (Full / FBM)
- Stock: `GET /inventories/{inventory_id}/stock/fulfillment` — current fulfillment stock (available/not_available). Operations: `GET /stock/fulfillment/operations/search?seller_id=&inventory_id=&date_from=&date_to=` (params `seller_product_id` CSV, `type`, `external_references.shipment_id`, `limit`, `sort`; default window 15 days). Operation record: `{id,seller_id,inventory_id/seller_product_id,date_created,type,detail{available_quantity,not_available_quantity,not_available_detail[]{status,quantity}},result{total,available_quantity,not_available_quantity,not_available_detail[]},external_references[]{type,value}}`. Paging uses `scroll`/`scroll_id`.
- Operation `type` values: inbound_reception, fiscal_coverage_ajustment (BR only), sale_confirmation, sale_cancelation, sale_delivery_cancelation, sale_return, withdrawal_reservation, withdrawal_cancelation, withdrawal_delivery, transfer_reservation, transfer_ajustment, transfer_delivery, quarantine_reservation, quarantine_restock, lost_refund, disposed_tained, disposed_expired, removal_reservation, STOCK_AUDIT, SALE_CANCELATION.
- Full shipments: `mode=me2`, `logistic_type=fulfillment`; statuses auto-transition, NO seller action, NO seller label printing. Inbound and logistic changes are done via Seller Center (front), not API — API is read-only for stock/operations. Notifications: `fbm_stock` / `fbm_stock_operations` (resource `/stock/fulfillment/operations/{id}`). Available in AR/BR/MX/CL/CO. Full invoicing (MLB, MLC) is automatic via ML biller.

### 7. MERCADO ENVIOS PLACES / agencies / pickup
- Places = `logistic_type=xd_drop_off` (seller drops at a Place/retail point → collection → HUB → carrier → buyer). Coleta = `cross_docking` (carrier collects at seller → HUB → carrier → buyer).
- Agency/drop-off point fields live in `shipping_address.agency{agency_id,carrier_id,description,type,open_hours,phone}`; address `types[]` may include `flex_pickup`.
- Agency delivery methods: `GET /sites/{site_id}/shipping_methods` → items with `deliver_to="agency"` (vs `"address"`), plus `company_id`, `company_name`, `id`, `name`, `type`, `status`.
- Dispatch preferences (currently `/marketplace/...`, initially Cainiao-only): `GET /marketplace/shipments/dispatch_preferences/carriers` → `{name,has_pickup,sites[],dispatch_info{pickup}}`; `PUT .../carriers/{name}` with `{pickup:true}` OR `{drop_off_agency:"TRAN_0001"}` (mutually exclusive); `GET/POST .../dispatch_preferences/pick_up_addresses`.

### 8. BILLING / FEES / COSTS
- **Fee simulator**: `GET /sites/{site}/listing_prices?category_id=&price=&currency_id=&listing_type_id=&logistic_type=&shipping_mode(s)=&billable_weight=` (tags=supermarket_eligible for supermarket). Response per listing_type: `currency_id`, `listing_type_id`, `listing_type_name`, `mapping`, `listing_exposure`, `requires_picture`, `free_relist`, `listing_fee_amount`, `listing_fee_details{fixed_fee,gross_amount}`, `sale_fee_amount`, `sale_fee_details{gross_amount,percentage_fee,meli_percentage_fee,fixed_fee,financing_add_on_fee}`, `stop_time`. **Critical**: `fixed_fee` now depends on `logistic_type`/`shipping_mode`/`billable_weight` — omit them and the computed fixed fee won't match what's charged. Price ≥ threshold → no fixed fee; below threshold, ME2 only charges fixed fee for Flex(self_service), while ME1/custom/not_specified always charge it.
- **Per-order fee detail (MLB)**: `GET /billing/integration/group/ML/order/details?order_ids=$ID1,$ID2` — billing reports filtered by orders/packs; MLB response includes detailed `sale_fee` composition (discounts, rebates per order).
- **Billing reports**: `GET /billing/integration/monthly/periods?group=MP|ML&document_type=BILL|CREDIT_NOTE&offset=&limit=` — returns the last 6 periods by default, max 12 via pagination; `document_type` required. `GET /billing/integration/periods/key/{YYYY-MM-01}/group/{ML|MP}/details?document_type=BILL` (period key = first day of month); `/summary`; `/documents`. `group` selects Mercado Libre vs Mercado Pago charges. Detail rows carry `charge_info{legal_document_number,legal_document_status,detail_id,detail_amount,transaction_detail,detail_type,detail_sub_type,concept_type}`, `discount_info{charge_amount_without_discount,discount_amount,rebate}`, `sales_info[]{order_id,operation_id,sale_date_time,transaction_amount,financing_fee,sale_fee{gross,net,rebate}}`, `warranty_info`, `prepaid_info`. (Best practice: avoid `offset` beyond 10,000 records — use the from_id method for complete integrity on large listings.)
- Order-level: `order_items[].sale_fee` (BR), `payments[].taxes_amount`, `payments[].shipping_cost`. Shipping cost to seller: `/shipments/{id}/costs` `senders[].cost`. Mercado Pago per-payment detail: `GET https://api.mercadopago.com/v1/payments/{id}`.

### 9. POST-SALE (Pós-venda) — claims v1 (post-purchase) + returns v2
- `/v1/claims/` deprecated (May 6 2024) → use `POST-PURCHASE`: `GET /post-purchase/v1/claims/search` filters: `stage`(claim/dispute/recontact/none), `status`(opened/closed), `type`, `resource`+`resource_id`, `reason_id`, `site_id`, `players.role`(complainant/respondent), `players.user_id`, `parent_id`, `date_created`, `order_id`, `pack_id`, `last_updated`, `range=date_created:after:...,before:...`, `sort=last_updated:asc|desc`. Recommend always including `players.user_id`+`players.role`.
- `GET /post-purchase/v1/claims/{id}` → `stage`, `status`, `type`, `resource`+`resource_id`, `reason_id`, `fulfilled`, `quantity_type`, `players[]{type(buyer/seller/receiver/sender/collector),role,user_id,available_actions[]{action,mandatory,due_date}}`, `related_entities[]`, `resolution`.
- `GET /post-purchase/v1/claims/{id}/detail` → `{due_date,action_responsible(seller/buyer/mediator),title,description,problem}`.
- `GET /post-purchase/v1/claims/{id}/status-history` → `[{stage,status,date,change_by}]`.
- `GET /post-purchase/v1/claims/{id}/affects-reputation` → `{affects_reputation, has_incentive}` (has_incentive=true → respond in 48h to avoid reputation impact).
- `GET /post-purchase/v1/claims/reasons/{reason_id}` → `{id,flow,name,detail,position,filter{group,site_id[]},settings{allowed_flows,expected_resolutions,rules_engine_triage},parent_id,status}`. Reason prefix identifies type: `PNR`=paid-not-received, `PDD`=defective product.
- `GET /post-purchase/v1/claims/{id}/expected-resolutions` → `[{player_role,user_id,expected_resolution(refund/return_product/product),status,details[]}]`. Actions: `POST .../expected-resolutions/allow-return`, `POST .../expected-resolutions/partial-refund` `{percentage}` (100% not allowed here), `POST /post-purchase/v1/claims/{id}/actions/open-dispute`.
- Messages: `GET /post-purchase/v1/claims/{id}/messages`; `POST /post-purchase/v1/claims/{id}/messages` or `/actions/message` `{receiver_role(complainant/mediator/respondent),message,attachments[]}`. Attachments: `POST /post-purchase/v1/claims/{id}/attachments` (form-data, ≤5MB, JPG/PNG/PDF/TXT); `GET .../attachments/{id}` and `/download`.
- **Returns v2**: `GET /post-purchase/v2/claims/{claim_id}/returns` → return object with `id` (=return_id), types `claim`/`dispute`/`automatic`, subtypes, statuses, `related_entities` (includes "reviews" when a review exists). Return review (unified): `POST /post-purchase/v1/returns/{return_id}/return-review` — empty body = OK; `[{reason,message,attachments[]}]` = failed (reasons e.g. SRF2, SRF7; check `apply` = package/order). Review evidence: `POST /post-purchase/v1/claims/{id}/returns/attachments`. Reviews list: `GET /post-purchase/v1/returns/{return_id}/reviews` → `resource_reviews[]{stage,status,product_condition,product_destination,reason_id,benefited,seller_status,seller_reason,missing_quantity}`. Return shipping cost: `GET /post-purchase/v1/claims/{id}/charges/return-cost?calculate_amount_usd=true`. Seller return-review availability = player `seller.available_actions` contains `return_review_ok`/`return_review_fail`. (CBT equivalents live under `/marketplace/v2/claims/...`.)

### 10. MESSAGING (post-sale)
- Get thread: `GET /messages/packs/{pack_id}/sellers/{seller_id}?tag=post_sale` (marks read; add `mark_as_read=false` to avoid). Response: `paging{limit,offset,total}`, `conversation_status{path,status,substatus,status_date,status_update_allowed,claim_id,shipping_id}`, `messages[]{id,site_id,client_id,from{user_id},to{user_id},status,subject,text,message_date{received,available,notified,created,read},message_moderation{status(clean/IN_MODERATION),reason,source,moderation_date},message_attachments[]{filename,original_filename,type,size,potential_security_threat,creation_date},message_resources[]{id,name},conversation_first_message,text_translated}`.
- Send: `POST /messages/packs/{pack_id}/sellers/{seller_id}?tag=post_sale` `{from{user_id},to{user_id},text,attachments[]}` (350-char limit; HTML `<a href>` allowed). Seller must initiate by choosing a reason. Messaging blocked on `cancelled` orders.
- Unread: `GET /messages/unread?tag=post_sale&role=seller|buyer` → `{user_id,results[]{resource,count}}` (≤500 conversations/call). Attachments uploaded first (site_id mandatory). CBT variant: `/marketplace/messages/packs/{order_id}` and `/marketplace/messages/unread`.

### 11. QUESTIONS (pre-sale, secondary)
- `GET /questions/search?item=&status=` supports `search_type=scan` + `scroll_id` for >1000 records. Answer via `POST /answers`. Notification topic `questions`.

### 12. FISCAL / INVOICING (Brazil / MLB)
- **Readiness gate**: `GET /can_invoice/items/{item_id}` (and `/can_invoice/items/{item_id}/variations/{variation_id}`) → `status=true` only when all fiscal data present (all variations must pass). This is what surfaces the red "Adicione os dados fiscais" state. At shipment level, `ready_to_ship` + substatus `invoice_pending` (and `awaiting_tax_documentation`) signal fiscal data is required before the label/dispatch.
- **When invoices are issued**: automatically ONLY in Mercado Envios Full via ML's integrated biller; in all other modalities the seller issues manually (or via the biller API).
- Fiscal data registration: `POST /items/fiscal_information/items` `{sku,item_id,variation_id}`; `PUT /items/fiscal_information/{sku_id}` with `tax_information{ncm,origin_type,origin_detail,tax_rule_id,csosn,cest,fci,ex_tipi,ean,med_anvisa_code,net_weight,gross_weight}` etc. Tax rules (Regime Normal): `GET/POST/PUT /users/{user_id}/invoices/tax_rules/{id}` (transactions→operations→rules, CFOP/CST/ICMS/IPI/PIS/COFINS; `product_origin_type` ∈ reseller/manufacturer/imported); operation messages: `/users/{user_id}/invoices/tax_rules/messages` (types item/fiscal/additional).
- Buyer fiscal data for NF-e: `GET /orders/{order_id}/billing_info` (header `x-version:2`) or `GET /orders/billing-info/{site_id}/{billing_info_id}` → `{site_id, buyer{cust_id, billing_info{name,last_name,identification{type(CPF/CNPJ),number},taxes{inscriptions{state_registration},taxpayer_type{description}},address{street_name,street_number,city_name,neighborhood,comment,state{name},zip_code,country_id},attributes{is_normalized,cust_type(CO/BU)}}}, seller{cust_id}}`. `invoice_type` field removed — integrator decides doc type from buyer identification.
- Upload/send NF-e XML (ME1/DropOff/Flex/Turbo): `POST /shipments/{id}/invoice_data?siteId=MLB` (XML), `PUT /shipment_invoice/{invoice_id}/?siteId=MLB`, `GET /shipments/{id}/invoice_data?siteId=MLB`. Send when shipment `ready_to_ship`+`invoice_pending`. **MLB: upload FORBIDDEN for fulfillment/cross_docking/xd_drop_off** (403 "you must use the biller of MercadoLibre"). Once label printed, NF-e cannot be altered; only validated (production, not homologação) XMLs accepted.
- Retrieve issued invoices: `GET /users/{user_id}/invoices/sites/MLB/batch_request/period/{YYYYMM}` (zip); `/period/stream?start=&end=&sale=all&return=all&full=all&others=all&file_types=xml,pdf&simple_folder=false`. Invoice types: SALE, SALE_RETURN, DEVOLUTION, INBOUND, INBOUND_RETURN, SYMBOLIC_INBOUND, SYMBOLIC_INBOUND_RETURN. Errors catalog: `GET /users/invoices/errors/MLB/{id}` → `{id,display_message,front_properties{correctedBy[],correctedIn}}`. Notification topic `invoices` (MLB, Full biller). Onboarding requires PJ + IE + A1 digital certificate.

### 13. REPUTATION & metrics
- `GET /users/{id}` includes `seller_reputation{level_id(e.g. "5_green"),power_seller_status(gold/platinum/null),real_level,protection_end_date,transactions{total,completed,canceled,period,ratings{positive,neutral,negative}},metrics{sales{period,completed},claims{period,rate,value,excluded{real_value,real_rate}},delayed_handling_time{period,rate,value,excluded{...}},cancellations{period,rate,value,excluded{...}}}}`. Protected sellers see real numbers under `excluded`. (Global-selling variant: `/global/users/seller_reputation`.)
- Green targets: cancellations ≤3%, delayed handling ≤10% (yellow ≤15%, orange ≤22%); handling window 3 days/72h. Period = last 60 days (BR needs 60 sales, else 365 days). Handling Time = ready_to_ship→shipped (cross_docking: in_hub→shipped).
- Order-level "afeta reputação": claim `/affects-reputation`; reputation-excluded orders carry `avoid_reputation` tag.

### 14. WEBHOOKS / notifications
- Configure callback URL + topics in the app manager. ML POSTs `{_id,resource,user_id,topic,application_id,attempts,sent,received}`; respond HTTP 200. **ML's stated rule**: a notification that has not received an HTTP 200 after the 8th retry (within ~1 hour) is considered lost — retrieve those via `GET /missed_feeds?app_id=`. (The commonly cited "reply within 500 ms" figure is a general webhook best practice, NOT stated in ML docs; ML only specifies the 8-retry/1-hour lost-feed rule.) Confirmed source IPs: **54.88.218.97, 18.215.140.160, 18.213.114.129, 18.206.34.84**. UTC timestamps.
- Relevant topics: `orders_v2` (recommended; resource `/orders/{id}`), `shipments` (`/shipments/{id}`), `messages` (created/read), `orders_feedback`, `claims`, `fbm_stock`/`fbm_stock_operations` (`/stock/fulfillment/operations/{id}`), `invoices` (MLB Full biller), `items`, `items_prices`, `stock_locations`, `flex_handshakes`, `public_offers`, `catalog_suggestions`. CBT namespace adds `marketplace_fbm_stock`, `marketplace_shipments`, `marketplace_questions`, `marketplace_items`. Any JSON change triggers a notification → always GET the resource and diff.

### 15. "Vendas" panel buckets → API mapping
- **Envios de hoje / Próximos dias**: `lead_time.estimated_handling_limit.date` (dispatch deadline) + shipment `status` in handling/ready_to_ship. Print-ready = `ready_to_ship` (+ label available); if `buffered`, use `buffering.date`.
- **Em trânsito**: shipment `status=shipped` (substatus out_for_delivery, in_transit, in_hub, etc.).
- **Finalizadas**: `status=delivered`.
- **Devoluções**: `not_delivered` (returning_to_sender etc.) + `/post-purchase/v2/claims/{id}/returns`.
- **Pós-venda**: `/post-purchase/v1/claims/search` + `/messages/unread`.
- **Coleta (pickup scheduling)**: dispatch/prep time `GET/PUT /users/{user_id}/service/carrier_pickup/processing_time_tool` (header `X-Version: v3`; only `SERVICE_TYPE=carrier_pickup`, covering cross_docking + xd_drop_off; multi-origin sellers with tag `warehouse_management` use `/nodes/{node_id}/service/carrier_pickup/processing_time_tool`; defaults 01:00 cross_docking / 01:30 xd_drop_off; PUT body `{processing_times{monday{processing_time:"01:00"},...}}`; GET fields per weekday `{modified_by_meli,intervention_type,visible,enabled,current_processing_time,available_options[]{processing_time,selected,highlight_level,disabled}}`); schedule `GET /users/{user_id}/shipping/schedule/{logistic_type}` → `schedule.{weekday}{work,detail[]{milkrun_same_day,from,to,cutoff,carrier{id,name},vehicle{id,license_plate,vehicle_type,only_for_today,new_driver},driver{id,name},sla,logistic_type}}`; capacity `PUT /users/{user_id}/capacity_middleend/{logistic_type}`. Coleta Rápida (BR, from Aug 2025) identified via `/shipments/{id}/sla` service=xd_same_day.

### 16. Rate limits / pagination / scopes
- `/orders/search`: 50/call; page via `offset`+date windows (no scan; `offset` hard-capped at 10,000 records — use date-window/from_id partitioning beyond that). CBT orders: 100 req/min. Flex: 1000 rpm. App: `max_requests_per_hour` (18000 default sample). `shipment_labels`/costs multiget: ≤50 ids. `/messages/unread`: ≤500 conversations/call. Multiget items/users: ≤20 ids. Billing offset max 10,000. Scopes: `read` for all GETs, `write` for label/tracking/message/claim/fiscal writes, `offline_access` for refresh tokens.

## Recommendations
1. **Ingest via webhooks, reconcile via search.** Subscribe to `orders_v2`, `shipments`, `claims`, `messages`, `fbm_stock`, `invoices`; on each event GET the resource and upsert. Nightly, reconcile with `/orders/search?seller=...&order.date_last_updated.from=...` in ≤50-record pages to catch missed feeds (also poll `/missed_feeds?app_id=`). Because notifications only carry a `resource` URL, always follow with a GET and diff against stored state.
2. **Branch all shipping logic on `shipment.logistic.type`.** fulfillment → read-only, ML prints labels & auto-invoices; drop_off → seller prints label + issues NF-e; cross_docking/xd_drop_off → seller prints label but NF-e via ML biller (upload forbidden); self_service → Flex windows/assignment endpoints. Always send `x-format-new: true` on shipment GETs.
3. **Wire the fiscal gate first (MLB).** Before showing "ready to ship", check shipment substatus `invoice_pending`/`awaiting_tax_documentation` and `/can_invoice/items/{id}`; surface the red "Adicione os dados fiscais" state and route to fiscal-data registration.
4. **Model fees from `/sites/{site}/listing_prices` with `logistic_type` + `shipping_mode` + `billable_weight`**, and reconcile actuals from `/billing/integration/.../order/details`. Never compute fixed fee without those params or the projection will diverge from actual charges.
5. **Bling/NF-e flow**: for non-Full modalities, pull buyer data from `/orders/{id}/billing_info` (x-version:2) into Bling, emit NF-e, then push XML via `/shipments/{id}/invoice_data?siteId=MLB` before label print. For Full, do NOT attempt upload (403) — consume ML-issued XML via the `invoices` topic + `/users/{id}/invoices/sites/MLB/batch_request` download and store back into Bling for reconciliation.
6. **Build the "Vendas" cockpit buckets** directly from `lead_time.estimated_handling_limit.date` + shipment `status`/`substatus` per Section 15, and add a Coleta/agendamento panel from `processing_time_tool` + `shipping/schedule/{logistic_type}`.
- **Thresholds that change the plan**: if per-page throughput needs exceed 50 orders/call, shard by date window (and remember offset's 10,000 ceiling); if reputation `delayed_handling_time.rate` approaches 10%, prioritize the "Coleta/Envios de hoje" bucket and tighten handling SLAs; if `cancellations.rate` nears 3%, gate seller-initiated auto-cancels; if `claims.rate` climbs, automate `/post-purchase` responses within the 48h `has_incentive` window.

## Caveats
- `/orders/search/recent` and `/orders/search/archived` are NOT documented — treat as non-existent; verify against a live token if needed.
- The full 7-value `order.status` enum (paid/confirmed/payment_required/payment_in_process/partially_paid/cancelled/invalid) isn't published on a single page; values are confirmed piecemeal across docs, plus `manually_cancelled` (buyer-search only).
- The "reply within 500 ms to notifications" figure is a general webhook best practice and is NOT stated in ML docs; ML's only documented rule is the 8-retry/~1-hour lost-feed window. Design for a fast 200 regardless.
- Dispatch-preferences pickup/agency endpoints are currently in the `/marketplace/...` (CBT) namespace, initially Cainiao-only — may not apply to a plain MLB seller token; verify live.
- Some field lists (notably CBT reputation and shipment examples) are drawn from `/marketplace/...` examples and may differ slightly from the plain MLB response; validate against a real MLB token.
- Flex driver GPS/live location is not exposed via API — only driver assignment (`/flex/.../assignment/v1`) and handshakes (`flex_handshakes`); live map tracking is only inside the Flex driver app.
- Billing `offset` is capped at 10,000 records; use the from_id method for complete integrity on large billing listings.
